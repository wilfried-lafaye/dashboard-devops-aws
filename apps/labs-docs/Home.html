Documentation – Quartz & Cloudflare Deployment

Author : Badr TAJINI - DevOps - ESIEE 2025-2026

Quickstart

  1. Copy the appendix A script into setup_quartz_cloudflare.sh at the repository root. Create this bash file if this is your first time.
  2. Copy the appendix B script into Makefile at the repository root. Create this makefile if this is your first time.
  3. Replace every CHANGE_ME_... placeholder (and set CLOUDFLARE_API_TOKEN). If you don’t own a domain in Cloudflare yet, leave DOMAIN set to CHANGE_ME_DOMAIN and the script will use the default .pages.dev URL. When you do want a custom hostname, point it at Cloudflare first and then set DOMAIN. Use EMAIL_DOMAIN for the comma-separated list of domains you plan to admit (e.g. esiee.fr,edu.esiee.fr).
  4. Bootstrap the site: make site/setup
  5. After editing content, redeploy: make site/update
  6. Optional lint before pushing: make site/check
  7. Copy .env.example to .env, fill in real values (Cloudflare account/token, GH user, etc.), and keep .env untracked; the deploy script auto-loads it.
  8. If Git isn’t initialized yet: git init -b main && git add . && git commit -m "Initial devops site" then git remote add origin <your-private-url> and git push -u origin main.
  9. Place secrets in a local env file (e.g. .env or ~/.devops-website.env) using the keys in .env.example before running setup_quartz_cloudflare.sh. Never commit the real values.

Prerequisites

  • macOS or Linux shell with bash, git, curl, and jq available.
  • Cloudflare account with Pages enabled.
  • GitHub CLI (gh) authenticated for the destination account.

Cloudflare API token

Create a custom token scoped to the Pages project (the domain you enter must already exist in your Cloudflare account):

  • Account → Cloudflare Pages : Edit
  • Zone → DNS : Edit
  • Account → Access: Apps and Policies : Edit

Before running the script, open the Zero Trust dashboard, go to Access → Overview, and click Enable Access. If you don’t see that button, create/activate your Zero Trust account (look for Start for free or create a first Access application). Until Access is enabled at least once, the API call will return 403 access.api.error.not_enabled and the site will stay public.

Enable Cloudflare Access for your account before running the script (Dashboard → Zero Trust → Access → Enable Access). Without it, the Access step will emit a warning and skip the restriction.

Deploy output

  • Repository root: ~/devops-website/devops-quartz-site
  • Static notebooks mirror: static/nb/
  • Build artifacts: public/
  • GitHub push: main branch (private), created if absent.

After the run

  • Update DNS records for the custom domain if needed.
  • Rotate the Cloudflare API token periodically and store it securely.
  • Remove local copies of sensitive screenshots or tokens.

Continuous integration & deployment

.github/workflows/site-ci.yml runs automatically on pushes and pull requests:

  • black --check on Python files under devops/ (skipped if none).
  • bash -n validation for the setup script.
  • Quartz build with public/ artifact upload.
  • Cloudflare Pages deploys from main when the repository secrets CLOUDFLARE_ACCOUNT_ID, CLOUDFLARE_API_TOKEN, and CLOUDFLARE_PROJECT are provided.
